# i18n (PT/EN) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add English (EN) support to Valuation Monitor while keeping Brazilian Portuguese (PT-BR) as default, without changing any URL paths.

**Architecture:** next-intl library with cookie-based locale detection (Accept-Language fallback). Flat JSON message files. PT/EN toggle in the header. No URL prefix — URLs stay exactly as they are.

**Tech Stack:** next-intl@latest, Next.js 16 App Router, cookie-based routing

---

## Decisions (from grill session)

| Decision | Choice |
|---|---|
| Library | `next-intl` |
| URL structure | Unchanged (no `/{locale}` prefix) |
| Language detection | Browser `Accept-Language` header |
| Manual override | Toggle PT/EN in header (cookie persisted) |
| Default locale | `pt-BR` |
| Message format | Flat JSON (`en.json`, `pt-BR.json`) |
| Preset keys | Stay in PT (e.g. "Limpar", "Bancos") — display string translated via `t()` |
| Not translated | Market names (BR FII, USA REIT), acronyms (DY, P/L, CAGR), brand (VALUATION MONITOR), URLs |

## Architecture

```
messages/
  pt-BR.json    ← default (all existing Portuguese strings)
  en.json       ← English translations
components/
  LanguageToggle.tsx  ← header switcher (writes cookie, refreshes)
```

### Data flow

1. User visits `/stocks/br`
2. next-intl middleware checks `NEXT_LOCALE` cookie
3. No cookie? Check `Accept-Language` header → `en` or `pt-BR`
4. No match? Default to `pt-BR`
5. React components use `useTranslations()` to render strings
6. User clicks PT/EN toggle → writes cookie → page refresh → middleware picks new locale
